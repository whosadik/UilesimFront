## Backend notes (ProfilePage)

### 1) `/api/me/favorite-category` — не хватает полей под текущий UI

Сейчас ответ даёт `favorite_category`, `window_days`, `profile_complete`, `explain`, но UI хочет ещё:

* `products_bought` (кол-во покупок/позиций)
* `total_spent` (сумма расходов за окно)

В текущем контракте этих полей нет. 

**Что сделать на бэке:**

* Расширить response `GET /api/me/favorite-category` (или сделать отдельный endpoint) и добавить:

  * `products_bought: int` (определить точную метрику: “кол-во покупок” = кол-во транзакций? или сумма qty по item’ам?),
  * `total_spent: decimal|string` (сумма `total_amount` по транзакциям в окне),
  * опционально `currency` (если в системе не одна валюта).
* Сохранить `explain` как есть (UI показывает “Как мы считаем?”).

### 2) Budget: фронт хочет min/max, бэк хранит enum

Сейчас профиль ожидает `budget` как enum (`low|medium|high`), а ProfileWizard на фронте оперирует `budgetMin/budgetMax`. 

**Варианты на бэке (выбрать один):**

* **Оставить как есть (рекомендую для MVP):** `budget` остаётся enum, а фронт маппит min/max -> enum.
* **Добавить расширение:** принять/отдавать `budget_range: {min, max}` дополнительно к `budget` (или вместо него), но это уже миграции/валидация/совместимость.

### 3) Оффер на странице: в UI сейчас статический

В API уже есть поддержка “следующего оффера” и клика по нему:

* `GET /api/me/next-offer` (возвращает assignment + offer + target + expires_at) 
* `POST /api/offers/click` 

**Что проверить/доделать на бэке (если будут баги в UI):**

* чтобы `expires_at` был всегда (или явно `null`), иначе фронт будет гадать формат;
* чтобы `offer.type/value` были стабильны (value часто decimal-string — это ок, но важно не менять тип поля). 

### 4) Рекомендации “Специально для вас”: формат ответа секциями

`GET /api/me/recommendations/home` возвращает `sections[]` (не плоский массив). 
Фронт сейчас уже пытается поддерживать оба варианта, но **лучше на бэке гарантировать один формат**.

**Что закрепить на бэке:**

* всегда возвращать `{ ok: true, sections: [...] }` (или хотя бы всегда `sections`), чтобы фронт не содержал лишние ветки.

### 5) Транзакции/покупки: если решишь считать агрегаты на бэке

Есть `GET /api/transactions/` (без глобальной пагинации). 
Если транзакций станет много — расчёт “любимой категории” и “total_spent” должен быть агрегатом на SQL уровне, а не “выгрузить всё и посчитать”.

**Итог по обязательному для этой страницы:**

* **обязательное**: расширить `/api/me/favorite-category` полями `products_bought` и `total_spent` (иначе текущий UI не заполнить данными из API). 
* остальное — опциональные улучшения/стабилизация контракта.
